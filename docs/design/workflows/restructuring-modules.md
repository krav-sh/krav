# Restructuring modules

## What

The developer says `the parser should actually be two separate modules`, `move the tokenizer under the lexer`, `merge these two modules`, or `reparent the auth module`. The module hierarchy changes, and the agent handles the downstream consequences: reassigning needs and requirements, reviewing task ownership, flagging traceability chain breaks.

## Why

Initial module hierarchies are hypotheses about the system's structure. They change as development reveals that the team drew boundaries wrong, that a component belongs elsewhere, or that two components should be one. Krav must support restructuring as a first-class operation, not a catastrophic rearrangement that breaks the graph.

## What happens in the graph

**Reparenting** is the simplest case. A module's `childOf` edge changes from one parent to another. The module keeps its needs, requirements, tasks, and test cases. But C-CROSS1 (derivation chain module alignment) might break if the module's requirements derive from needs on the old parent. The agent checks for these violations and reports them.

**Splitting** means creating new child modules and redistributing the parent's content. Requirements, needs, tasks, and test cases that belong to the original module need reassignment to the appropriate child. The agent proposes a distribution based on the content's scope and asks the developer to confirm.

**Merging** means moving all content from one module to another and removing the empty module. Needs, requirements, and tasks get their `module` field updated. Test cases that verify moved requirements keep their `verifies` edges. The agent deletes the empty module (it must have no children per the delete constraint).

In all cases, the agent checks for broken constraints after the restructuring, including C-CROSS1 (derivation alignment), C-CROSS2 (verification alignment), and C-CROSS3 (allocation targets). It reports violations and proposes fixes.

## Trigger patterns

`Split the parser into lexer and parser`, `move module X under module Y`, `merge these two modules`, `reparent X`, `the architecture needs restructuring`.

## Graph before

An existing module hierarchy with content (needs, requirements, tasks, test cases) attached to the modules under restructuring.

## Graph after

Changed `childOf` edges. Possibly reassigned `module` ownership on needs, requirements, tasks, and test cases. Possibly new suspect links or defects if the restructuring reveals inconsistencies.

## Agent interaction layer

### Skills

The `krav:restructure` skill builds this workflow. It has `disable-model-invocation: true` in its frontmatter because restructuring has broad graph impact and shouldn't happen without explicit developer intent. Preprocessing loads the current module hierarchy, the content attached to modules under restructuring (needs, requirements, tasks, test cases), and any cross-module constraint violations that would result from the proposed change.

The skill's instructed commands handle reparenting, splitting, and merging operations through the CLI. For splits, the skill instructions guide the agent through proposing a content distribution (which requirements go to which new child) and confirming with the developer before executing. For merges, the instructions handle content consolidation and empty module cleanup.

### Policies

The `baseline-protection` policy is critical when restructuring involves baselined modules. Moving content out of a baselined module or reparenting it modifies the protected scope, which the policy flags or blocks. The developer may need to run `krav moduleunlock` with justification before restructuring can proceed.

The `mutation-feedback` policy fires after each structural change (reparenting, content reassignment), keeping the agent aware of the changing hierarchy as the restructuring unfolds. The `graph-integrity` and `cli-auto-approve` policies ensure all changes go through the CLI. Restructuring may also generate suspect links on downstream edges, which surfaces in subsequent `session-context` injections as pending triage work.

### Task types

Restructuring doesn't create tasks directly, but its consequences often produce work. Suspect links generated by content reassignment feed into the `krav:suspect` workflow. Constraint violations (C-CROSS1, C-CROSS2, C-CROSS3) surface as defects that produce `remediate-defect` tasks. If the restructuring reveals that the new module hierarchy needs architecture work, `decompose-module` and `define-interface` tasks may follow.

## Open questions

**How disruptive is restructuring to in-progress work?** If tasks are in progress for a module that the developer splits, those tasks need reassignment. The agent should block restructuring of modules with in-progress tasks, or at minimum warn loudly.

**Suspect link implications.** When a requirement moves from one module to another, should its upstream `derivesFrom` edges become suspect? The requirement's statement hasn't changed, but its context has. The INCOSE answer would be yes: any change to a node's key properties should trigger suspect marking on downstream edges. But module reassignment feels different from statement modification.

**Baseline interaction.** If a module has an approved baseline and the developer then restructures it, the baseline still references the old state. The baseline itself doesn't break (it's anchored to a commit), but any assumptions about "what's baselined" need revisiting. Should restructuring automatically create a defect noting that the baseline predates the restructuring?

**Integration module updates.** If a module under reparenting had `integrates` edges from an integration module, those edges might need updating. The agent should check for `integrates` references to the restructured module.
